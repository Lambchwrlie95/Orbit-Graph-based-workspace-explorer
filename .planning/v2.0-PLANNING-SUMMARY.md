## Rationale: Why Phases, Not Waves?

v2.0 is structured as **4 sequential phases** rather than parallel waves because:

1. **User-Facing Completeness**: Each phase delivers a complete, testable capability
2. **Natural Dependencies**: 
   - Phase 6 needs Phase 5 (desktop integration)
   - Phase 6 Grid view needs Phase 7 thumbnails
   - Phase 8 Enhanced Inspector needs Phase 7 image analysis
3. **Integration Points**: Each phase deeply integrates with the v1.0 foundation
4. **Testing Boundaries**: Each phase is independently shippable

## Architecture Decisions (v2.0)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **Editor** | Monaco Editor | Industry standard, VS Code compatible |
| **Packaging** | AppImage primary | Single-file, no installation, Tauri native |
| **Duplicate Detection** | Perceptual hash (pHash) | Finds similar images, not just exact matches |
| **Color Extraction** | K-means / color-thief | Industry standard, Rust ecosystem support |
| **Thumbnail Storage** | `~/.local/share/orbit/thumbnails/` | XDG compliant, user-local |

## Relationship to v1.0

v2.0 builds directly on the v1.0 foundation:

- **Phase 1-4** (v1.0): Core infrastructure, explorer, graph, performance
- **Phase 5-8** (v2.0): Desktop polish, enhanced browsing, assets, code editing

v1.0 proved the core concept: graph-first file intelligence. v2.0 makes it a complete desktop application that users can install, use for daily file work, and optionally edit code within.
